---
title: Secret Scan
description: Surface repository secret risk and whether raw values can reach a detected agent.
---

`fida scan` is the risk lens for Fida's secret-leak prevention promise. It surfaces where secrets exist, whether they are tracked by git, and whether a raw value can still reach a detected agent. It is local-only and never emits a secret value, its length, or any fragment of it.

## Scan for risk

`fida scan` inspects a repository for secret exposure and reports a single risk level.

```bash
fida scan
fida scan --path ./service
fida scan --json
fida scan --fail-on high
fida scan --include-ignored
fida scan --exclude vendor
fida scan --mcp
```

What it checks per discovered file:

| Signal               | Meaning                                                                                   |
| -------------------- | ----------------------------------------------------------------------------------------- |
| Sensitive file       | Name matches `.env`, `.env.*`, `*.pem`, `*.key`, `id_rsa`, or `id_ed25519`.               |
| Hardcoded secret     | A provider key or private-key block embedded in a regular source file (e.g. `sk_live_…`). |
| Content match        | The secret scanner found a secret pattern (files up to 5 MiB are scanned).                |
| Git tracked          | The file is tracked by git (committed exposure).                                          |
| Protection           | The weakest detected agent path for the file: `enforced`, `best_effort`, `incomplete`, or `inactive`. |
| Raw secret exposure  | Whether the raw value could reach a model instead of a redacted or blocked view.          |
| Legacy compatibility | JSON still includes older fields such as `policy_covered` and `agent_reachable`.          |

Risk levels:

- `high` — a secret is tracked by git, or `raw_secret_exposure` is true for any detected agent path that is not enforced.
- `medium` — a secret exists, but the model-bound path is enforced/redacted or blocked.
- `low` — no sensitive files, content matches, or hardcoded secrets (including an empty scan).

Default-excluded directories (`.git`, `target`, `node_modules`, `dist`, `build`, `.next`) and their subtrees are never traversed. Git-ignored ordinary files are skipped for performance, but sensitive filenames such as `.env`, `.env.*`, private keys, and PEM files are still scanned because they are exactly the files an agent should never see raw. Use `--include-ignored` when you want the broader ignored tree too. A missing or unreadable `--path` exits non-zero before any traversal. With `--fail-on high`, a high-risk result exits non-zero so the scan fits in CI.

### Protection-aware output

Human output and JSON distinguish file access from model exposure. An agent may be able to read a file through Fida and still be safe when the value returned to the model is redacted. The important field is `raw_secret_exposure`.

In JSON, newer fields appear alongside the old shape:

```json
{
  "risk": "medium",
  "protection": "enforced",
  "raw_secret_exposure": false,
  "findings": [
    {
      "path": ".env",
      "sensitive": true,
      "protection": "enforced",
      "raw_secret_exposure": false,
      "policy_covered": false,
      "agent_reachable": true
    }
  ]
}
```

The legacy fields remain for consumers that already parse them, but new integrations should key off `protection` and `raw_secret_exposure`.

### What the content scan recognizes

Content matching reuses the same detector Fida applies everywhere — including at the gateway, so a redacted read benefits from it too. Out of the box it recognizes `.env` `KEY=value` assignments, PEM private-key blocks, and a built-in catalog of high-precision provider key formats (AWS, GitHub, Google, Slack, Stripe, OpenAI, Anthropic, JWT), plus any patterns you add to your policy's `secrets` section.

Regular source files (anything not matching a sensitive name) are scanned too, so a key hardcoded in code — `const key = "sk_live_…"` — is surfaced as a **hardcoded secret** finding. To avoid noise, source scanning skips the broad `KEY=value` heuristic, so ordinary assignments like `version = "1.2.3"` are not flagged; the precise provider catalog, private-key blocks, and your policy patterns still apply. Files over 5 MiB and binary files are skipped.

The catalog favors precision over recall: each pattern is anchored to its provider's known shape to keep false positives low. Fida is **not** a replacement for a dedicated secret-history scanner — see [Comparisons](/docs/reference/comparisons) for how it complements tools like gitleaks and GitGuardian.

### MCP risk scan

`fida scan --mcp` (alias `--agents`) reads the MCP configurations of installed agents (Claude, Cursor, Codex, OpenCode, Windsurf, GitHub Copilot) read-only and flags risky servers — filesystem, shell, browser, git, sqlite, postgres, docker, kubernetes, or secrets — recommending they be routed through the Fida gateway. Missing configs are skipped; unparseable configs are reported as non-fatal errors and the scan continues.

## Tighten the policy from real activity

`fida observe` runs an agent in observe mode and records its observed actions; `fida policy suggest` turns them into a least-privilege allowlist.

```bash
fida observe -- codex
fida policy suggest
fida policy suggest --write
```

The suggested policy's allow rules correspond exactly to the recorded observed actions. Regardless of what was observed, model-bound redaction stays on, sensitive writes are denied, and network access, package installation, and `git push` ask before running. `fida policy suggest` shows a line-by-line diff against the current policy before any write; `--write` applies it atomically, leaving the current policy unchanged on failure.
