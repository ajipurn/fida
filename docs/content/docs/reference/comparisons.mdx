---
title: Comparisons
description: How Fida compares to secret scanners, and where it fits alongside context/token tools like LeanCTX, RTK, and Caveman.
---

Fida belongs in the same workflow as secret scanners and context/token tools, but it solves a different problem: it keeps raw secret values out of AI-agent context. Policy and audit still matter, but the MVP promise is model-bound redaction before a secret reaches the model.

## Secret scanners (gitleaks, GitGuardian, trufflehog)

Dedicated secret scanners are pattern-and-history engines: they carry hundreds of detectors and scan commits, diffs, and history to find committed or leaked credentials. Fida is **not** trying to out-detect them, and you should keep using one.

The difference is _when_ and _where_ they act:

|             | Secret scanners                 | Fida                                                        |
| ----------- | ------------------------------- | ----------------------------------------------------------- |
| Primary job | Find secrets in code/history    | Stop raw secret values from reaching an AI model at runtime |
| Acts at     | Commit / CI / history           | Agent setup, file reads, shell output, audit/report writing |
| Output      | A list of findings to remediate | Redacted safe views, protection status, scan risk, audit    |
| Coverage    | Broad detector catalog          | A precise built-in catalog, focused on live redaction       |

Use both: a scanner keeps secrets out of history; Fida keeps a live agent from receiving the raw secrets you still have on disk. `fida scan` is a fast protection-aware risk check, not a replacement for a full history scan.

## Where Fida fits alongside context and token tools

LeanCTX, RTK, and Caveman are token- and context-economy tools: they shape how much an agent reads, how much terminal output it ingests, and how verbosely it replies. Fida sits on a different axis — it controls whether raw secret values enter that context at all — so it is not an alternative to any of them. They stack.

### Summary

| Tool    | Primary job                | Controls                                           |
| ------- | -------------------------- | -------------------------------------------------- |
| Fida    | Secret leak prevention     | Whether raw secret values can reach model context. |
| LeanCTX | Context intelligence layer | What agents read and remember.                     |
| RTK     | CLI output compression     | How much terminal output reaches context.          |
| Caveman | Response style compression | How many tokens the agent spends talking.          |

The other three optimize the token economy; Fida is the secret-protection layer beneath them. RTK can shrink `cargo test` output, but Fida is the layer that redacts any secret that surfaces in an allowed read or shell output. Advanced policy can also decide whether commands like `curl` or `cat .env` run at all.

## Recommended stack

```text
Caveman -> terse agent responses
RTK -> compressed terminal output
LeanCTX -> compact context and memory
Fida -> secret redaction, protection status, scan risk, audit
```

The products are complementary. Fida should be positioned as secret leak prevention, not as a token saver or a general firewall first.
